# 代码构建和修复行为规范

## 核心原则：重构式修改优先于增量式补丁

### 一、修改分类定义

#### ❌ 禁止：增量式修改（补丁思维）
- 简单地在现有代码中添加if判断来"绕过"问题
- 为了解决一个bug而单独写一段特殊处理逻辑
- 在函数中直接返回硬编码值来临时解决问题
- 使用try-catch吞掉错误而不分析根本原因
- 复制粘贴相似代码而不是抽象通用逻辑

#### ✅ 要求：重构式修改（根因思维）
- 遇到问题时先分析根本原因，从架构层面解决
- 将重复代码抽象为通用组件或工具函数
- 建立统一的错误处理和数据校验机制
- 重构现有逻辑使其自然地支持新需求
- 每次修改都要考虑对整体代码结构的影响

### 二、具体执行规范

#### 1. 问题定位流程
```
发现bug/需求 → 追踪根因 → 分析影响范围 → 设计重构方案 → 实施修改 → 验证完整性
```

#### 2. 修改前必做
- [ ] 理解现有代码的整体架构和业务逻辑
- [ ] 定位问题的根本原因而非表象
- [ ] 评估修改对其他模块的影响
- [ ] 设计从根源解决问题的方案

#### 3. 修改后必查
- [ ] 确认修改消除了根本原因，而非仅仅隐藏了症状
- [ ] 检查是否有相似逻辑需要同步重构
- [ ] 确认代码更简洁、更易于理解和维护
- [ ] 运行完整构建和测试确保无回归

#### 4. 代码质量标准
- 单一职责：每个函数/组件只做一件事
- 开闭原则：对扩展开放，对修改封闭
- DRY原则：不要重复自己（Don't Repeat Yourself）
- YAGNI原则：你不会需要它（避免过度设计）

### 三、典型场景处理

#### 场景1：登录验证失败
- ❌ 增量式：在登录函数里添加特殊判断，如果静态部署就跳过验证
- ✅ 重构式：
  - 分析项目有哪几种部署模式
  - 设计统一的认证策略抽象层
  - 让不同部署模式实现同一接口
  - 确保代码在所有模式下都优雅运行

#### 场景2：API返回数据格式不一致
- ❌ 增量式：在每个调用处做数据转换和兼容处理
- ✅ 重构式：
  - 建立统一的数据模型定义
  - 在API层做数据标准化转换
  - 统一所有接口的返回格式

#### 场景3：相似的组件代码重复
- ❌ 增量式：复制粘贴后小改一下
- ✅ 重构式：
  - 提取共性为父组件或通用组件
  - 使用props/配置来实现差异
  - 建立组件库确保一致性

### 四、违规检查清单

每次提交代码前自查：
1. 我的修改是否解决了根本问题？
2. 是否有更好的架构设计方案？
3. 我的修改是否引入了重复代码？
4. 我的修改是否让代码更难理解和维护？
5. 是否有类似的代码需要同步修改？

### 五、奖励与惩罚

- **奖励**：优秀的重构式修改将被标记为最佳实践
- **警告**：发现增量式补丁将要求重新设计方案
- **禁止**：严重破坏代码结构的修改将被拒绝合并

---

**注意**：本规范适用于所有代码修改，包括但不限于bug修复、功能新增、性能优化、代码重构等项目。任何修改都必须遵循"从根本解决问题"的原则，禁止简单打补丁的方式绕过问题。
